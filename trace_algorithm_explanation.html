Détection des bords de trace événementielle

Cette page explique l’algorithme actuel de détection de la trace dans la vue Trace : accumulation des événements, projection dans le repère de trace, détection des bords par support local, construction des lignes haut/milieu/bas, mesure de largeur 3D, puis régression pondérée optionnelle de la trajectoire.

1. Vue générale du pipeline

L’algorithme temps réel commence par une fenêtre mobile qui suit la balle dans l’image. Cette fenêtre est stabilisée par une parabole spatiale y(x) ajustée sur les centres détectés de la balle. Dans les événements accumulés par cette fenêtre, le code ne cherche pas directement un cercle : pour une balle rapide, l’image événementielle produit plutôt une trace allongée. Le code cherche donc un ruban dense, puis estime ses deux bords et son axe central.

But

Extraire une bande cohérente dans un nuage événementiel très bruité.

Principe

Transformer le problème 2D en coupes 1D perpendiculaires à la trace.

Sortie

Trois courbes lissées et une largeur mesurée normalement à la courbe centrale.

Carte du code actuel

Tout le calcul décrit ici se trouve dans src/Ball_Tracking_Cpp/src/TraceAnalysis.cpp (API dans TraceAnalysis.hpp), sans dépendance UI ; Gui.cpp orchestre et dessine. La vue trace est principalement un pipeline de visualisation et d’estimation géométrique. Les fonctions importantes sont les suivantes.

DrawTraceScene()

Point d’entrée de la vue Trace. Récupère les événements, appelle le fit du ruban, dessine les courbes et calcule la pose 3D.

SetTraceMotionWindow()

Reçoit la position courante de la balle et la parabole image y(x), puis définit la fenêtre mobile de suivi.

Accumule uniquement les événements qui tombent dans cette fenêtre mobile, avant le fit des bords.

BuildTracePointsFromFloatSource()

Exécute l’analyse Trace même quand la vue affichée n’est pas Trace, pour garder la pose 3D bleue à jour.

point = centre + direction × s + normale × h

C’est la représentation interne utilisée pour transformer le nuage 2D en coupes perpendiculaires.

width = high - low

La largeur locale est mesurée sur l’axe normal, puis réorientée sur la normale de la ligne du milieu.

PCA : estimation de la direction

La PCA, ou analyse en composantes principales, sert ici à trouver l’orientation du nuage d’événements. On ne l’utilise pas pour détecter les bords directement : elle sert à construire un repère initial aligné avec la trace.

L’idée est simple : si les événements forment une trace allongée, leur dispersion est beaucoup plus grande dans le sens de la trace que dans le sens de la largeur. La PCA cherche précisément la direction où la variance des points est maximale.

1. Centre moyen du nuage

Après un premier rejet des points très éloignés, le code calcule le centre moyen des points restants : 

meanX = moyenne des x
meanY = moyenne des y

2. Matrice de covariance

Ensuite, on mesure comment les points se dispersent autour de ce centre : 

dx = x - meanX
dy = y - meanY

cxx = moyenne(dx * dx)   // variance horizontale
cxy = moyenne(dx * dy)   // corrélation x/y
cyy = moyenne(dy * dy)   // variance verticale

C = [ cxx  cxy ]
    [ cxy  cyy ]

3. Axe principal

On cherche ensuite l’eigenvecteur associé à la plus grande eigenvaleur de cette matrice. Cet eigenvecteur est la direction où le nuage est le plus étiré.

λ₁ = plus grande eigenvaleur de C
direction ≈ eigenvecteur associé à λ₁
normale = (-direction.y, direction.x)

Le code utilise toujours un PCA global, puis ajoute des PCA locaux par fenêtres de temps. Le slider PCA ms fixe la durée cible de ces fenêtres, ce qui reste stable quand la trace accumulée devient plus longue.

4. PCA temporels toutes les N millisecondes

Les événements retenus sont découpés par timestamp en fenêtres d’environ PCA ms. Chaque fenêtre produit son propre centre et sa propre direction PCA. 

pour chaque tranche temporelle:
  points_t = événements dont le timestamp est dans la tranche
  pca_t = PCA(points_t)
  centre_t = moyenne des points_t
  direction_t = axe principal local

Ces directions locales sont ensuite alignées pour éviter l’ambiguïté de signe du PCA, puis combinées avec le PCA global. L’orientation initiale devient un mélange de :

45 % direction PCA globale
35 % moyenne pondérée des directions PCA temporelles
20 % direction entre le premier et le dernier centre temporel

Les petits segments bleus affichés sur la trace montrent les directions PCA temporelles. Ils servent à vérifier visuellement si le découpage temporel suit bien la trace.

Même avec plusieurs PCA, le repère s/h reste un repère initial. La courbure fine est toujours récupérée ensuite par les fits locaux des lignes haut/milieu/bas.

2. Passage dans le repère de trace

Après le filtrage grossier, une PCA donne la direction principale de la trace. On définit ensuite deux axes :

s = position le long de la trace
h = position perpendiculaire à la trace

Chaque événement est transformé depuis l’image caméra vers ce repère local. Cela permet de découper la trace en tranches le long de s, puis de chercher les bords dans une simple distribution 1D sur h. 

delta = point - centre
s = dot(delta, direction)
h = dot(delta, normale)

C’est une étape clé : au lieu de chercher une forme complexe dans l’image, on cherche une zone dense dans chaque coupe perpendiculaire.

3. Détection locale des bords

Pour chaque tranche en s, on récupère toutes les valeurs h. Le bord est supposé être un changement brutal de densité : peu ou pas d’événements dehors, beaucoup d’événements dans la trace.

L’algorithme actuel ne prend pas simplement un percentile pour dire “bord haut” et “bord bas”. La fonction active est EstimateSupportedEdges() : elle cherche les extrêmes qui ont assez de voisins dans un rayon local sur l’axe h.

L’ancien algorithme Density par histogramme a été retiré. Il n’y a donc plus de bouton Edge, plus de Hist bins, et plus de Density %. Le réglage se fait maintenant avec les paramètres propres au support.

Mode Support : trouver les extrêmes soutenus

Les valeurs h finies sont triées. Le code cherche ensuite le premier point bas et le dernier point haut qui ont plusieurs voisins proches dans un rayon fixe. Un point isolé très loin ne suffit donc pas à créer un bord.

Exemple concret : imaginons une coupe locale où la vraie trace occupe environ h = -6 px à h = +6 px. Quelques événements isolés apparaissent à -12 px et +13 px. Un simple minimum/maximum choisirait ces deux outliers comme bords. Le mode Support les ignore parce qu’ils n’ont pas assez de voisins proches. 

values = sort(finite h)
localSupport = clamp(count / Support div, Support min, Support max)
supportRadius = Support radius px

exemple :
  count = 36 valeurs
  Support div = 28
  Support min = 3
  Support max = 9
  localSupport = clamp(36 / 28, 3, 9) = 3
  supportRadius = 1.75 px

low = premier h qui a au moins 3 valeurs dans [h, h + 1.75]
high = dernier h qui a au moins 3 valeurs dans [h - 1.75, h]

correction = clamp(rawWidth * Border %, 0, 1.5 px)
low  -= correction
high += correction

Le balayage bas part du plus petit h. Pour chaque candidat, il compte combien de valeurs triées tombent dans une fenêtre de 1.75 px vers la droite. Le premier candidat qui atteint localSupport devient low. Le balayage haut fait la même chose depuis la droite, avec une fenêtre de 1.75 px vers la gauche.

Le mode Support répond à la question : “où commencent et finissent les points vraiment soutenus par le paquet d’événements ?”. C’est souvent le meilleur choix pour une trace nette avec des bords pleins.

Étape 1 : coupe brute sur l’axe h
Étape 2 : balayage bas avec fenêtre de support
Étape 3 : balayage haut depuis la droite
Étape 4 : correction, largeur et validation

La petite correction appliquée à low et high compense le fait que les événements mesurés sont des centres de pixels. Le code agrandit légèrement la largeur avec une correction bornée, proportionnelle à la largeur brute.

Validation de la coupe

Une coupe est acceptée seulement si elle a assez d’événements finis, si high reste strictement au-dessus de low, et si le nombre d’événements compris dans la largeur finale atteint le support minimal demandé. 

supportCount = nombre de h entre low et high
reject si finiteValues < max(minSupportCount, 8)
reject si rawWidth < 2 px
reject si supportCount < minSupportCount
reject si high <= low
La largeur locale vaut high - low. La position du milieu local vaut (low + high) / 2. Cette largeur est ensuite utilisée comme diamètre apparent pour reconstruire la profondeur.

4. Construction des lignes haut, milieu, bas

Après la détection des bords dans chaque tranche, le code possède une suite de petits échantillons locaux : 

pour chaque bin valide :
  s        = position de la tranche
  hLow     = bord local inférieur
  hMiddle  = milieu local
  hHigh    = bord local supérieur

Ces échantillons sont filtrés par cohérence spatiale. Si un point fait une pointe isolée par rapport à une prédiction locale de ses voisins, il est rejeté. C’est le filtrage qui évite les pics sur la ligne verte ou magenta.

Ensuite on ajuste trois modèles locaux indépendants :

  • courbe du milieu à partir des hMiddle ;
  • courbe du haut à partir d’un des deux bords ;
  • courbe du bas à partir de l’autre bord.

L’ordre du fit est réglé par Line order : ordre 1 pour une ligne locale, ordre 2 pour une courbe locale.

5. Mesure de largeur perpendiculaire

La largeur orange est calculée après la construction de la ligne du milieu. Pour chaque échantillon de largeur, le code prend un point sur la courbe jaune et calcule sa tangente locale. La normale est ensuite imposée comme direction de mesure. 

middle = point sur ligne jaune
tangent = tangente locale de la ligne jaune
normal = perpendiculaire(tangent)
largeur = distance entre les deux bords sur cette normale

Même si les intersections avec les courbes haut/bas tombent à des positions légèrement décalées, on projette leurs offsets sur la normale de la ligne jaune, puis on reconstruit les extrémités sur cette même normale.

Résultat : les segments orange sont perpendiculaires à la ligne jaune par construction.

Cette largeur est aussi celle utilisée pour l’estimation 3D, selon :

Z = f_eff × diamètre_balle / largeur_px
erreur pour 1 px ≈ Z / largeur_px

6. Régression pondérée de trajectoire

Le bouton Weighted reg active une variante de régression pour la trajectoire 3D. Elle est utilisée pour la trajectoire classique issue du circle fitting, et aussi pour la trajectoire bleue issue de la trace. Le but est de garder une courbe réactive : les mesures récentes comptent plus que les anciennes, et les points loin de la courbe actuelle comptent moins.

Modèle ajusté

Le modèle géométrique ne change pas. Seuls les poids des observations changent.

x(t) = a t + b
y(t) = a t + b
z(t) = a t² + b t + c

En mode non pondéré, chaque point a le même poids. En mode pondéré, chaque point reçoit un poids total :

poids_total = poids_récence × poids_robuste

Poids de récence

Le poids de récence dépend de la position temporelle du point dans l’historique courant. Le point le plus récent a un poids proche de 1. Les points anciens décroissent exponentiellement. 

age = clamp((newestTime - t) / timeRange, 0, 1)
poids_récence = exp(-3 * age)

Cette formule garde les derniers points dominants sans supprimer totalement l’historique. Un vieux point peut encore stabiliser la courbe s’il est cohérent.

Poids robuste contre les outliers

Le code fait d’abord une régression normale. Ensuite il mesure le résidu de chaque point par rapport à cette première courbe. Le résidu est comparé à une échelle robuste basée sur la médiane des résidus. 

résidu = abs(y_mesuré - y_prédit)
scale = max(1e-6, median(residus) * 1.4826)
poids_robuste = 1 / (1 + (résidu / scale)²)

Un point proche de la courbe garde presque tout son poids. Un point très loin de la courbe voit son poids diminuer fortement, au lieu de tirer toute la trajectoire.

Où c’est appliqué

Dans le tracker classique, la pondération est appliquée dans Update3DTrack() après l’ajout d’une nouvelle pose 3D. Dans la vue 3D, la trajectoire bleue Trace est recalculée avec le même principe avant d’être dessinée et avant d’afficher ses équations. 

fit_initial = régression normale
poids = récence(t) * robustesse(résidu)
fit_final = régression pondérée
La pondération ne remplace pas la détection de bords. Elle stabilise seulement la trajectoire finale quand quelques poses 3D sont anciennes ou aberrantes.

7. Paramètres et effets

Les paramètres de la vue Trace n’agissent pas tous au même niveau. Certains changent la quantité d’événements disponible, d’autres changent le découpage, d’autres changent le lissage.

Paramètre Effet principal Si on augmente Si on diminue
Window ms Durée d’accumulation temporelle. Plus d’événements, trace plus dense, mais plus de flou temporel. Plus réactif, mais moins robuste si peu d’événements.
Trace ms Durée de mémoire de la trace temps réel accumulée. Plus d’ancienne trace dans le fit, mais plus de calcul. Plus réactif et léger, mais moins de contexte derrière la balle.
Follow window px Demi-longueur de la fenêtre mobile qui suit la balle dans l’image. Capture plus largement la trace autour de la parabole y(x), mais laisse passer plus de bruit. Filtre plus fort les points isolés, mais peut couper la trace si le suivi ou la parabole est en retard.
Bin width px Largeur cible des tranches le long de la trace. Tranches plus larges, plus robustes mais moins locales. Tranches plus fines, plus précises mais moins d’événements par tranche.
Support div Divise le nombre d’événements de la coupe pour calculer le support demandé. Demande moins de voisins, accepte plus facilement les extrêmes, mais laisse passer plus d’outliers. Demande plus de voisins, rejette mieux les points isolés, mais peut rentrer les bords vers le centre.
Support min Minimum de voisins exigés localement, même pour une coupe pauvre. Rejette plus fort les outliers, mais peut supprimer des coupes peu denses. Accepte les coupes pauvres, mais les extrêmes deviennent plus fragiles.
Support max Maximum de voisins exigés localement dans les coupes très denses. Rend les bords plus stricts sur les grosses coupes. Évite qu’une coupe dense demande trop de voisins et perde un vrai bord fin.
Support radius px Rayon de voisinage sur l’axe normal h. Connecte des points plus espacés, plus tolérant au bruit d’échantillonnage. Exige un paquet compact, plus strict contre les parasites mais plus fragile si le bord est troué.
Border % Correction extérieure appliquée après les bords soutenus. Mesure plus large, plus proche du bord apparent ; trop haut sous-estime la profondeur. Mesure plus proche des centres d’événements ; trop bas surestime la profondeur.
Local window Rayon de voisinage du fit local des lignes. Lignes plus lisses, moins de bruit. Suit mieux les courbes rapides, mais peut devenir nerveux.
Line order Forme locale du modèle. 2 suit mieux les arcs. 1 est plus stable et plus droit.
Width step px Espacement cible entre deux mesures orange. Moins de mesures, calcul plus léger. Plus de mesures de largeur, meilleure couverture mais plus de calcul.
Edge refine Seconde passe de détection des bords dans une bande étroite autour des courbes prédites. ON (défaut) : les extrêmes soutenus sont re-détectés par bin dans [bord prédit ± marge], les calottes d'extrémité (~1 rayon de balle à chaque bout, où les coupes mesurent des cordes du disque et non le diamètre apparent) sont exclues, puis les trois courbes sont refittées — les événements isolés hors ruban ne tirent plus les bords et les extrémités ne biaisent plus la 3D. OFF : une seule passe, l'ancien comportement.
Width fit Lissage robuste du profil de largeur avant la conversion 3D. Smooth remplace chaque largeur par un polynôme robuste largeur(t) ajusté sur toute la trace (IRLS, pondéré par le support local) : la profondeur balistique varie de façon lisse, donc le diamètre apparent aussi, et le bruit pixel par échantillon est moyenné. Raw garde les largeurs locales brutes, l'ancien comportement, utile comme référence.
PCA ms Durée cible de chaque PCA temporel. Fenêtres plus longues, direction plus stable. Fenêtres plus courtes, suit mieux les changements rapides mais demande plus d’événements.
Radius gate Accumule seulement les clusters validés par le filtre rayon/RMS. ON garde une trace plus propre. OFF accumule les événements filtrés sans cette validation, utile pour comparer.
Polarity Filtre les événements par polarité avant le fit de trace. Negative est le meilleur réglage actuel sur events_filtered.h5. All sert au diagnostic si une séquence change de convention.
Weighted reg Active la régression pondérée par récence et par distance à la courbe. Trajectoire plus réactive, moins influencée par les poses anciennes ou aberrantes. Trajectoire strictement moyenne, utile comme référence de test.
Fit input Choisit les points raw ou undistorted. Undist est généralement meilleur pour la géométrie 3D. Raw peut parfois être plus proche de l’image native.

À ces valeurs, 1 pixel d’erreur sur la largeur produit environ 7.0 cm d’erreur de profondeur.

Support moyen estimé : 795 événements par tranche, pour une trace exemple de 330 px. Le minimum interne demandé par tranche vaut environ 79 événements avant détection de bord.

Avec ces réglages, EstimateSupportedEdges() demande 9 voisins dans un rayon de 1.75 px sur l’axe h.

Pour une largeur brute exemple de 21 px, Border % ajoute environ 0.74 px de chaque côté, donc 1.47 px au diamètre apparent.

8. Diagnostic visuel

Cette table sert à relier ce que l’on voit dans l’interface avec la partie de l’algorithme qui est probablement en cause.

Symptôme Cause probable Action utile
La ligne verte ou magenta fait une pointe isolée. Une tranche locale valide un extrême avec trop peu de support réel, ou le fit local a trop peu de voisins. Augmenter Support min, baisser Support div, augmenter Local window, ou augmenter un peu Bin width px.
Les deux bords sont bons mais la ligne jaune n’est pas au centre. Les bords haut/bas sont cohérents, mais le fit du milieu est trop lissé ou influencé par quelques bins. Passer Line order à 2 si la trace est courbe ; baisser légèrement Local window.
La largeur orange n’atteint pas visuellement les lignes verte/magenta. La largeur est forcée sur la normale de la ligne jaune ; les intersections de bord sont projetées sur cette normale. Vérifier la tangente de la ligne jaune. Si elle est mauvaise localement, la normale sera mauvaise aussi.
La profondeur 3D saute alors que les lignes ont l’air correctes. La profondeur dépend de 1 / largeur_px. Une erreur de 1 px peut valoir plusieurs centimètres. Stabiliser la largeur, réduire les points aberrants, puis envisager un filtre de Kalman sur centre + largeur.
La fin de trace part n’importe où. Il n’y a plus assez de support local ; le fit extrapole ou suit du bruit. Le code coupe déjà la plage sMin/sMax sur les bins valides. Réduire Trace ms peut aider.
Les bords sont trop à l’intérieur de la trace. Le support demandé est trop strict, le rayon de support est trop petit, ou la correction extérieure est trop faible. Augmenter Support div, augmenter légèrement Support radius px, ou augmenter Border %.

9. Pseudo-code complet

Voilà la version compacte de l’algorithme, proche de la structure du code. Les noms ne sont pas exactement tous les noms C++, mais l’ordre logique est le même. 

function detectTrace(events):
  points = getTracePoints(events)
  if points.count < 500:
    return invalid

  robustCenter = median(x), median(y)
  inliers = removeFarRadialOutliers(points, robustCenter)
  if inliers.count < 250:
    return invalid

  globalDirection = PCA_major_axis(inliers)
  temporalDirections = PCA_by_time_slices(inliers, PcaSlices)
  direction = combine(globalDirection, temporalDirections)
  normal = perpendicular(direction)

  projections = []
  for p in inliers:
    s = dot(p - center, direction)
    h = dot(p - center, normal)
    projections.push(s, h, p, time)

  bins = splitByS(projections, LineBins)

  lowSamples = []
  middleSamples = []
  highSamples = []

  for bin in bins:
    edges = EstimateSupportedEdges(bin.h, SupportSettings)
    // extrêmes qui ont assez de voisins locaux sur h
    if edges.valid:
      lowSamples.push(bin.s, edges.low)
      middleSamples.push(bin.s, edges.middle)
      highSamples.push(bin.s, edges.high)

  removeSpatialSpikes(lowSamples, middleSamples, highSamples)

  middleCurve = localFit(middleSamples, LocalWindow, LineOrder)
  lowCurve = localFit(lowSamples, LocalWindow, LineOrder)
  highCurve = localFit(highSamples, LocalWindow, LineOrder)

  topCurve, bottomCurve = sortByImageY(lowCurve, highCurve)

  widths = []
  for sampleTime in WidthSamples:
    s = localMedianS(sampleTime)
    middle = middleCurve(s)
    tangent = tangentOf(middleCurve, s)
    normal = perpendicular(tangent)
    width = measureDenseWidthAlongNormal(middle, normal)
    widths.push(width)

  removeWidthSpikes(widths)
  return topCurve, middleCurve, bottomCurve, widths

Pseudo-code de la régression pondérée

function weightedFit(times, values, model):
  initial = unweightedFit(times, values, model)
  residuals = abs(values - predict(initial, times))
  scale = max(1e-6, median(residuals) * 1.4826)

  for each sample i:
    age = clamp((newestTime - times[i]) / timeRange, 0, 1)
    recentWeight = exp(-3 * age)
    robustWeight = 1 / (1 + (residuals[i] / scale)^2)
    weight[i] = recentWeight * robustWeight

  return weightedLeastSquares(times, values, weight, model)

Pourquoi cette approche marche

La caméra événementielle ne donne pas une image pleine : elle donne des changements. Sur une balle rapide, les événements forment naturellement une trace avec une densité élevée à l’intérieur et un changement net sur les bords.

La bonne information n’est donc pas le point extrême, mais la transition entre zone peu dense et zone dense.

Pourquoi éviter le percentile

Un percentile répond à la question “où sont les 96 % des points ?”. Ce n’est pas exactement la question du bord. Si quelques points isolés existent dehors, ou si la densité est asymétrique, le percentile peut être décalé.

Le code actuel répond plutôt à : “où commence et finit le paquet vraiment soutenu par des voisins locaux ?”. C’est plus robuste contre un point isolé sur les côtés.

Limite principale

Si le paquet dense principal lui-même est faux, par exemple en fin de trace avec peu de support, le détecteur local peut encore se tromper. C’est pour cela qu’on ajoute la cohérence spatiale et qu’un Kalman ou un smoother pourra ensuite aider.

10. Conseils de réglage

Si les bords ont des pics isolés

Augmenter Support min, baisser Support div, ou augmenter légèrement Local window.

Si les bords sont trop lissés

Diminuer Local window ou passer Line order à 2 pour mieux suivre la courbure.

Si les bords ne collent pas au bord réel

Vérifier Support radius px et Border %. Si les bords sont trop dedans, augmenter légèrement l’un des deux.

Si la profondeur saute

La largeur en pixels est très sensible : à 21 px, 1 px d’erreur vaut environ 7 cm à 1.47 m.

Si la fin de trace est mauvaise

L’algo coupe déjà la zone sans support suffisant. Une fenêtre temporelle plus courte peut aussi réduire la queue bruitée.

Si la ligne orange semble inclinée

Elle est reconstruite sur la normale de la ligne jaune. Si elle semble fausse, c’est souvent la tangente de la ligne jaune qui est localement mal estimée.
